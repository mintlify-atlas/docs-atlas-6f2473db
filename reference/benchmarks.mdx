Parlor includes two benchmark scripts in `src/benchmarks/`. Use them to measure latency on your hardware or against a remote server.

<Note>
  Both scripts use `uv run` and must be run from the `src/` directory where `pyproject.toml` lives.
</Note>

## bench.py — end-to-end benchmark

`bench.py` connects to the running Parlor server over WebSocket and measures real end-to-end latency, including network round-trip, LLM inference, and TTS synthesis.

### Prerequisites

Start the server before running the benchmark:

```bash
uv run server.py
```

### Running

```bash
uv run python benchmarks/bench.py
```

To benchmark a remote server, set `SERVER_URL`:

```bash
SERVER_URL=ws://192.168.1.10:8000/ws uv run python benchmarks/bench.py
```

### What it tests

The benchmark runs three sections in sequence:

<AccordionGroup>
  <Accordion title="Individual turns">
    Opens a fresh WebSocket connection for each test case (fresh conversation context each time). Tests six input combinations:

    | Test | Payload |
    |---|---|
    | Text only | Plain text string |
    | Audio 2s | 2-second synthetic WAV (440 Hz sine wave) |
    | Audio 5s | 5-second synthetic WAV |
    | Image only | 320×240 solid-colour JPEG |
    | Image + Audio 2s | Both image and 2s audio |
    | Image + Audio 5s | Both image and 5s audio |
  </Accordion>
  <Accordion title="Multi-turn conversation">
    Keeps a single WebSocket connection open across five turns, so the model accumulates conversation history. Each turn sends an image together with audio of increasing duration (1s, 2s, 3s, 5s, 2s).
  </Accordion>
  <Accordion title="Correctness checks">
    Sends specific inputs and asserts expected outputs:

    - **Tool called** — verifies the model used the `respond_to_user` tool and returned a transcription
    - **Has response** — verifies the response text is non-empty
    - **No raw delimiters** — verifies the model's internal `<|"|>` separator tokens are stripped before sending
    - **Image described** — verifies an image-only turn returns a description longer than 10 characters
    - **Audio processed** — verifies an audio-only turn returns a non-empty response
  </Accordion>
</AccordionGroup>

### Output columns

```
Test                   LLM     TTS   Total  Response
--------------------------------------------------------------------------------
Text only            1.84s   0.31s   2.17s  Sure! Here's a fun fact about the ocean...
Audio 2s             2.01s   0.44s   2.48s  I heard you say hello! How can I help?...
```

| Column | Description |
|---|---|
| `LLM` | Time from receiving the user message to completing LLM inference |
| `TTS` | Total time to synthesise all audio for the response |
| `Total` | Wall-clock time from sending the message to receiving `audio_end` |
| `Response` | First 50 characters of the AI's text response |

## benchmark_tts.py — TTS backend comparison

`benchmark_tts.py` benchmarks the Kokoro TTS backend(s) in isolation, without a running server. On Apple Silicon it compares `mlx-audio` (Apple GPU) against `kokoro-onnx` (CPU). On other platforms it benchmarks `kokoro-onnx` only.

### Running

```bash
uv run python benchmarks/benchmark_tts.py
```

<Note>
  This script loads TTS models directly and does not require the server to be running.
</Note>

### What it measures

The script benchmarks three sentence lengths — short (~30 chars), medium (~110 chars), and long (~350 chars) — with 2 warmup runs and 5 measured runs each. For each combination it reports:

| Metric | Description |
|---|---|
| Mean | Average synthesis time across 5 runs (±stdev shown inline) |
| Min | Fastest single run |
| Audio | Duration of the generated audio in seconds |
| RTF | Real-time factor — synthesis time ÷ audio duration. Values below `1.0` mean faster than real-time |
| SR | Sample rate of the generated audio in Hz |

On Apple Silicon, the script also runs a streaming benchmark (`mlx-audio` with `stream=True`) and reports time-to-first-chunk (TTFC) for each sentence length, then prints a speedup comparison between the two backends.

### Forcing a specific backend

On Apple Silicon, set `KOKORO_ONNX=1` to benchmark only the ONNX backend:

```bash
KOKORO_ONNX=1 uv run python benchmarks/benchmark_tts.py
```

## Reference numbers (Apple M3 Pro)

| Stage | Time |
|---|---|
| Speech + vision understanding | ~1.8–2.2s |
| Response generation (~25 tokens) | ~0.3s |
| Text-to-speech (1–3 sentences) | ~0.3–0.7s |
| **Total end-to-end** | **~2.5–3.0s** |

Decode speed: ~83 tokens/sec on GPU (Apple M3 Pro).
